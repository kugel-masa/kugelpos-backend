# カテゴリプロモーション機能仕様書

## 概要

商品カテゴリに対して期間限定の割引プロモーションを設定できる機能。ショッピングカートに商品を追加する際、システムが商品のカテゴリと有効なプロモーションルールに基づいて自動的に割引を適用する。

本機能は拡張可能なプロモーション基盤の第一弾として実装される。将来的に他のプロモーションタイプ（商品単位、数量ベースなど）を追加できるアーキテクチャを前提とする。

## 課題

現在のPOSシステムにはカテゴリレベルでプロモーションキャンペーンを実施する機能がない。店舗運営者は以下のことができない：
- 特定の商品カテゴリに対して期間限定の割引キャンペーンを作成する
- 会計時に自動的にプロモーション割引を適用する
- 異なる有効期間を持つ複数のプロモーションを同時に管理する
- プロモーションの販促効果を測定・分析する

これによりマーケティングの柔軟性が制限され、販売時点での手動割引適用が必要となっている。

## ユーザーシナリオとテスト

### シナリオ1: 店舗管理者がカテゴリプロモーションを作成する

**アクター**: 店舗管理者

**事前条件**:
- 店舗管理者が認証済み
- 商品カテゴリがシステムに存在する

**フロー**:
1. 店舗管理者がプロモーション管理画面にアクセスする
2. 店舗管理者が以下の情報で新規プロモーションを作成する：
   - プロモーションタイプ：「カテゴリ割引」を選択
   - プロモーション名と説明
   - 対象店舗（例：「渋谷店、新宿店」または「全店舗」）
   - 対象カテゴリ（例：「飲料」）
   - 割引率（例：10%オフ）
   - 開始日時と終了日時
3. システムがプロモーションデータを検証する
4. システムがプロモーションを有効として保存する

**期待結果**: プロモーションが作成され、指定された期間中に自動的に適用される。

**受入条件**:
- 必須フィールドをすべて入力してプロモーションを作成できる
- システムが同一カテゴリの重複プロモーションを防止する（設定可能）
- システムが日付範囲を検証する（終了日は開始日より後でなければならない）

### シナリオ2: 顧客が自動割引を受ける

**アクター**: 顧客（レジ担当者経由）

**事前条件**:
- カテゴリに対する有効なプロモーションが存在する
- 現在時刻がプロモーションの有効期間内である

**フロー**:
1. レジ担当者がプロモーション対象カテゴリの商品をスキャンする
2. システムが商品のカテゴリが有効なプロモーションに一致することを検出する
3. システムが自動的に割引を計算して適用する
4. 割引がプロモーションコード付きの明細割引として表示される
5. 顧客が割引価格を確認する

**期待結果**: 手動介入なしに割引が自動的に適用される。

**受入条件**:
- 割引が同一取引内で適用される（遅延なし）
- 割引金額が正しく計算される
- 割引がプロモーション名付きでレシートに表示される
- 「割引制限」フラグが設定された商品は割引されない
- **プロモーションコードが取引ログに記録される**

### シナリオ3: プロモーションの期限切れ

**アクター**: システム

**事前条件**:
- 終了日が設定されたプロモーションが存在する
- 現在時刻が終了日を過ぎている

**フロー**:
1. 顧客が以前プロモーション対象だったカテゴリの商品を購入する
2. システムが有効なプロモーションをチェックする
3. システムが有効なプロモーションを見つけない（期限切れ）
4. 商品が通常価格で追加される

**期待結果**: プロモーション期限切れ後は割引が適用されない。

**受入条件**:
- 期限切れのプロモーションは適用されない
- プロモーションが無効な場合にエラーやユーザー混乱が発生しない
- 過去の取引は適用された割引を保持する

### シナリオ4: 複数のプロモーションが一致する

**アクター**: システム

**事前条件**:
- 複数の有効なプロモーションが同じ商品に適用可能
- プロモーションに異なる割引率が設定されている

**フロー**:
1. レジ担当者が複数のプロモーション対象の商品をスキャンする
2. システムが一致するすべてのプロモーションを特定する
3. システムが各プロモーションの割引後価格を計算する
4. システムが最も安くなるプロモーション（最大の割引率）を選択する
5. システムが1つのプロモーションのみを適用する

**期待結果**: 顧客にとって最も有利な（最安値になる）プロモーションが適用される。

**受入条件**:
- 商品ごとに1つのカテゴリプロモーションのみが適用される
- 最大の割引率を持つプロモーションが自動選択される
- 顧客は常に最安値で購入できる

### シナリオ5: 店舗管理者がプロモーション実績を確認する

**アクター**: 店舗管理者

**事前条件**:
- プロモーションが実施済みで取引が発生している

**フロー**:
1. 店舗管理者がプロモーション実績画面にアクセスする
2. 店舗管理者が対象プロモーションと期間を選択する
3. システムがプロモーションコードで取引を集計する
4. システムが以下の実績を表示する：
   - 適用回数（取引数、明細数）
   - 割引総額
   - 対象売上金額

**期待結果**: プロモーションの販促効果を数値で確認できる。

**受入条件**:
- プロモーションコード単位で実績を集計できる
- 期間指定で絞り込みができる
- 正確な割引総額が表示される

## 機能要件

### FR-1: プロモーションマスタデータ管理

| ID | 要件 | 受入条件 |
|----|------|----------|
| FR-1.1 | システムはプロモーションタイプを識別できる | タイプフィールドでプロモーション種別を区別できる |
| FR-1.2 | システムはカテゴリプロモーションの作成を許可する | 必須フィールドをすべて入力してプロモーションを保存できる |
| FR-1.3 | システムはプロモーションコード、名前、説明を保存する | すべてのフィールドが永続化され取得可能 |
| FR-1.4 | システムは1つ以上のカテゴリのターゲット設定をサポートする | 複数のカテゴリコードを関連付けできる |
| FR-1.4.1 | システムは対象店舗の指定をサポートする | 店舗コードを指定でき、未指定時は全店舗適用 |
| FR-1.5 | システムはパーセント割引をサポートする | 割引率が正しく計算される |
| FR-1.6 | システムはプロモーションの開始・終了日時を強制する | プロモーションは有効期間内のみ適用される |
| FR-1.7 | システムはプロモーションの有効化/無効化をサポートする | 無効化されたプロモーションは適用されない |
| FR-1.8 | システムは複数プロモーション一致時に最安値を選択する | 最大の割引率を持つプロモーションが適用される |
| FR-1.9 | システムはプロモーションの論理削除をサポートする | 削除されたプロモーションは適用されないが履歴として保持される |

### FR-2: 自動割引適用

| ID | 要件 | 受入条件 |
|----|------|----------|
| FR-2.1 | システムはカートに商品追加時に有効なプロモーションをチェックする | すべての商品追加でチェックが実行される |
| FR-2.2 | システムは商品カテゴリを有効なプロモーションと照合する | category_codeフィールドを使用して照合する |
| FR-2.2.1 | システムは現在の店舗がプロモーション対象かを確認する | 店舗コードを照合、または全店舗対象を確認 |
| FR-2.3 | システムはプロモーションルールに基づいて割引を計算する | 割引額 = 価格 × 数量 × 割引率 |
| FR-2.4 | システムは明細に自動的に割引を適用する | 割引がline_item.discountsに表示される |
| FR-2.5 | システムは商品の割引制限フラグを尊重する | is_discount_restricted=trueの商品はスキップされる |
| FR-2.6 | システムは割引詳細にプロモーションコードを記録する | プロモーションコードが追跡可能 |

### FR-3: プロモーション検索

| ID | 要件 | 受入条件 |
|----|------|----------|
| FR-3.1 | システムはすべてのプロモーションを一覧表示するAPIを提供する | ページネーションされたリストを返す |
| FR-3.2 | システムは有効なプロモーションを取得するAPIを提供する | 現在時刻とis_activeフラグでフィルタリングする |
| FR-3.3 | システムはカテゴリ別にプロモーションを取得するAPIを提供する | target_category_codesでフィルタリングする |
| FR-3.4 | システムはタイプ別にプロモーションを取得するAPIを提供する | promotion_typeでフィルタリングする |
| FR-3.5 | システムは店舗別にプロモーションを取得するAPIを提供する | target_store_codesでフィルタリングする |

### FR-4: 販売実績へのプロモーション記録

| ID | 要件 | 受入条件 |
|----|------|----------|
| FR-4.1 | システムは取引ログにプロモーションコードを記録する | 適用されたプロモーションが取引データに含まれる |
| FR-4.2 | システムは明細単位でプロモーションコードを保持する | 各明細の割引にプロモーションコードが紐付く |
| FR-4.3 | システムはプロモーションコードで取引を検索できる | プロモーション実績の集計が可能 |

### FR-5: プロモーション実績集計

| ID | 要件 | 受入条件 |
|----|------|----------|
| FR-5.1 | システムはプロモーションコード単位で適用回数を集計できる | 取引数・明細数が取得可能 |
| FR-5.2 | システムはプロモーションコード単位で割引総額を集計できる | 割引金額の合計が取得可能 |
| FR-5.3 | システムは期間指定でプロモーション実績を絞り込める | 開始日・終了日で範囲指定可能 |

## 成功基準

| 基準 | 目標 | 測定方法 |
|------|------|----------|
| プロモーション作成成功率 | 有効データで100% | 必須フィールド入力時にエラーなし |
| 割引適用精度 | 100%正確な計算 | 自動テストによる検証 |
| 割引適用遅延 | ユーザーに感知されない | 商品スキャン時に顕著な遅延なし |
| レシートでのプロモーション表示 | 100% | 適用されたすべてのプロモーションが名前付きで表示 |
| 取引ログへの記録率 | 100% | すべての適用プロモーションが取引ログに記録される |
| 実績集計の正確性 | 100% | 集計結果と個別取引の合計が一致 |
| 100件の同時プロモーション処理 | パフォーマンス低下なし | 複数の有効プロモーションでの負荷テスト |

## 主要エンティティ

### プロモーション（汎用）

| 属性 | 説明 |
|------|------|
| promotion_code | プロモーションの一意識別子 |
| promotion_type | プロモーションタイプ（例: "category_discount"） |
| name | プロモーションの表示名 |
| description | 詳細説明 |
| start_datetime | プロモーションが有効になる日時 |
| end_datetime | プロモーションが期限切れになる日時 |
| is_active | プロモーションが有効かどうか |

### カテゴリプロモーション詳細

| 属性 | 説明 |
|------|------|
| target_store_codes | 対象店舗コードのリスト（空の場合は全店舗適用） |
| target_category_codes | このプロモーションが適用されるカテゴリコードのリスト |
| discount_rate | 割引率（パーセント、例: 10 = 10%オフ） |

### 割引情報（取引ログ内）

| 属性 | 説明 |
|------|------|
| promotion_code | 適用されたプロモーションのコード |
| promotion_type | プロモーションタイプ |
| discount_rate | 適用された割引率 |
| discount_amount | 割引金額 |

### 関連

- プロモーションは1つ以上のカテゴリをターゲットにする（category_code経由）
- プロモーションはカート内の明細に適用される
- 適用された割引は元のプロモーションコードを参照する
- 取引ログはプロモーションコードを保持し、実績集計に使用される

## 設計方針

### 拡張可能なプロモーション基盤

プロモーションシステムは将来の拡張を前提として設計する：

1. **プロモーションタイプの分離**: 共通属性（コード、名前、期間、有効フラグ）と タイプ固有属性（カテゴリコード、割引率など）を分離
2. **プラグインアーキテクチャ**: cartサービスの既存sales_promoプラグイン構造を活用し、新しいプロモーションタイプをプラグインとして追加可能に
3. **タイプ識別子**: promotion_typeフィールドで処理ロジックを切り替え

### 将来追加可能なプロモーションタイプ（参考）

| タイプ | 説明 | 今回のスコープ |
|--------|------|----------------|
| category_discount | カテゴリ単位の割引 | **対象** |
| item_discount | 商品単位の割引 | スコープ外 |
| quantity_discount | 数量ベース割引 | スコープ外 |
| bundle_discount | セット割引 | スコープ外 |

## 前提条件

1. **商品ごとに単一プロモーション**: 明細ごとに1つのカテゴリベースプロモーションのみが適用される。プロモーションの重複適用はスコープ外。
2. **最安値選択**: 複数のプロモーションが一致する場合、最も割引率が高い（最安値になる）プロモーションが自動選択される。
3. **パーセント割引のみ**: 割引タイプはパーセント（％）のみをサポート。固定金額割引はスコープ外。
4. **テナント分離**: プロモーションはテナント固有（マルチテナンシーサポートが必要）。
5. **タイムゾーン処理**: プロモーション時刻はテナントのローカルタイムゾーンで保存・比較される。
6. **割引制限の優先**: 商品レベルの割引制限は常にプロモーションルールより優先される。
7. **遡及適用なし**: プロモーションは既存のカート/取引に遡及的に適用されない。
8. **手動割引との共存**: カテゴリプロモーションは同一明細の手動適用割引と共存できる。
9. **実績集計の基盤**: 取引ログに記録されたプロモーションコードを使用して実績集計を行う。

## スコープ外

- 固定金額割引（パーセント割引のみサポート）
- 顧客固有のプロモーション（ロイヤルティプログラム）
- 数量ベースのプロモーション（X個買うとY個無料）
- バンドルプロモーション（複数商品セット）
- クーポンコード入力
- 店舗固有のプロモーションターゲティング（将来の拡張）
- 高度な分析ダッシュボード（基本的な集計のみ対応）

## 依存関係

- カテゴリマスタ（master-dataサービス）- 存在必須
- category_codeフィールドを持つ商品マスタ（master-dataサービス）- 存在必須
- カートサービスの割引システム - 明細割引をサポート必須
- セールスプロモプラグインアーキテクチャ（cartサービス）- 存在するが未使用
- 取引ログ（journalサービス）- プロモーションコード記録先
- レポートサービス - 実績集計機能の実装先

## リスク

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| プロモーションチェックによるパフォーマンス影響 | 会計処理の遅延 | 有効プロモーションを短いTTLでキャッシュ |
| 誤った割引計算 | 金銭的損失 | 包括的な自動テスト |
| プロモーション有効期間でのタイムゾーン混乱 | 誤った適用時間 | 明確なドキュメントとUTC保存・ローカル変換 |
| 取引ログの肥大化 | ストレージ・クエリ性能の低下 | プロモーション情報の最小化、インデックス最適化 |
